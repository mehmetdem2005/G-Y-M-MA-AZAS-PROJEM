ÖRNEK DETAYLANDIRMA: P-SEC-01 (SQL Enjeksiyonu Engelleme)

Alan Değer
Politika ID P-SEC-01
Başlık SQL Enjeksiyonuna Karşı Sıfır Tolerans
Kategori Güvenlik
Şiddet CRITICAL
Sorumlu Ajanlar BackendCoder, DBAgent, CodeReviewer, SecurityScanner
Uygulandığı Protokoller Code_Generation_v2, SAST_Scan_Request_v2
Kural Tanımı LLM tarafından üretilen veya herhangi bir kaynaktan gelen kod içerisinde, kullanıcı girdisi (request parameter, form input, vb.) ile bir SQL sorgu metninin string birleştirme (+, concat, format) yoluyla oluşturulması yasaktır.
Deterministik Doğrulama SecurityScanner AST analizi ile şu kalıpları arar:   - "SELECT..." + request.getParameter(...)   - f"SELECT * FROM users WHERE id = {user_input}"   - String.format("SELECT...", userInput)
İhlal Durumu FSM State: CONTESTED

Muafiyet (Waiver) Politikası İzin Verilmez. Bu politika için WAIVED durumu tanımlı değildir. Hiçbir koşulda SQL enjeksiyonuna izin verilemez.
Denetim Kanıtı - code_snippet_hash: İhlale neden olan kodun SHA-256 değeri   - cwe_id: CWE-89   - source_location: Dosya ve satır numarası   - audit_trail_ref: WORM log kaydı referansı

POLİTİKA YÖNETİM VE İHLAL İŞ AKIŞI

Politika Değerlendirme Sırası (Her Eylem İçin)

```
1. Ajan bir eylem başlatır (LLM çağrısı, kod üretimi, tool kullanımı).
2. İlgili hook (`before_llm_call`, `on_tool_call`, vb.) tetiklenir.
3. `PolicyEnforcer` ajanına `Policy_Evaluation_Request_v1` gönderilir.
   - `subject`: Eylemi yapan ajan
   - `action`: Yapılmak istenen işlem
   - `resource`: İşlemin hedefi
   - `context`: Ek bağlam
4. `PolicyEnforcer`, ilgili tüm politikaları Rego motorunda değerlendirir.
5. Sonuç:
   - `PERMIT`: Eylem devam eder.
   - `DENY`: Eylem engellenir, `FSM State: CONTESTED`.
   - `DENY_WITH_WAIVER`: Muafiyet mekanizması devreye girer.
6. İhlal durumunda `ProcedureEngine` ilgili prosedürü başlatır.
7. Tüm süreç `AuditTrailWORM`'a kaydedilir.
```

Muafiyet (Waiver) Yönetim Prosedürü

Bir politika ihlali için muafiyet talep edildiğinde:

1. Talep, politikanın waiver_policy.allowed alanı kontrol edilir.
2. allowed: false ise talep otomatik reddedilir.
3. allowed: true ise:
   · Talep, approver_agent olarak belirtilen ajana iletilir.
   · Onaylayıcı ajan (veya insan operatör) gerekçeyi değerlendirir.
   · Onaylanırsa, benzersiz bir waiver_id oluşturulur.
   · Muafiyet, max_duration_hours süresince geçerlidir.
   · Süre dolduğunda muafiyet otomatik kalkar, ihlal yeniden değerlendirilir.
4. Tüm muafiyetler AuditTrailWORM'a özel bir WAIVER_GRANTED kaydı ile işlenir.
